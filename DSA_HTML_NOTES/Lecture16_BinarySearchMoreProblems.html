Problem 1 : Find Minimum in Rotated Sorted Array [LeetCode 153]

Problem Statement Given: Ek array jisme unique integers hain, pehle ascending order mein sorted tha, phir kisi unknown pivot par rotate kar diya gaya.
Task: Us array ka minimum element return karna hai.
Constraint: Solution O(log n) time mein chalna chahiye.
Examples
InputOutputExplanation
[3,4,5,1,2]1Array 5 ke baad rotate hua. 1 sabse chota.
[4,5,6,7,0,1,2]00 minimum element hai.
[11,13,15,17]11Array rotate nahi hua. First element minimum.
[2,1]1Rotation ke baad 1 minimum ban gaya.

Core Intuition

Intuition / Insight Rotated sorted array ko samjho: ek sorted array ko kisi pivot se rotate karne se do sorted halves ban jaate hain. Jaise [4,5,6,7,0,1,2] mein [4,5,6,7] left sorted half hai aur [0,1,2] right sorted half hai.

Key Observation: Agar nums[mid] > nums[end] hai, toh minimum hamesha right half mein hoga. Kyunki mid se end tak ek "drop" hai — wahi rotation point hai. Agar nums[mid] < nums[end] hai, toh minimum mid ya uske left mein hoga.

Yeh property hume har step pe half search space discard karne deti hai.
Rotated Sorted Array Structure
4 5 6 7 0 1 2
Left Sorted Half → MINIMUM (Pivot) → Right Sorted Half

Approach 1 : Brute Force (Linear Scan)

Workflow / Algorithm
  1. First element ko minimum assume karo.
  2. Array ko traverse karo.
  3. Har element ko current minimum se compare karo.
  4. Chota element milne pe update karo.
  5. End mein minimum return karo.
// Brute Force: O(n)
class Solution {
public:
    int findMin(vector<int>& nums) {
        int minimum = nums[0];
        for (int num : nums) {
            minimum = min(minimum, num);
        }
        return minimum;
    }
};
Complexity Analysis
MetricValueReason
TimeO(n)Har element ek baar check hota hai
SpaceO(1)Single variable extra space
Why Not Brute Force? Array ki sorted structure ka fayda nahi uthaya. Binary Search se har comparison ke baad half search space discard kar sakte hain. O(n) se O(log n) mein optimize hota hai.

Approach 2 : Optimal (Binary Search)

Key Insight Compare nums[mid] with nums[end]:
  • nums[mid] > nums[end] → Ek "drop" mid aur end ke beech mein hai. Minimum right side mein hoga. start = mid + 1.
  • nums[mid] < nums[end] → Right side sorted hai. Minimum mid ya left mein hoga. end = mid.
Condition: while(start < end) — jab start == end, wohi minimum element ka index hai.
Algorithm Steps
  1. start = 0, end = n - 1 initialize karo.
  2. while (start < end) loop chalao.
  3. mid = start + (end - start) / 2 calculate karo.
  4. nums[mid] ko nums[end] se compare karo:
    • nums[mid] > nums[end] → start = mid + 1 (right mein jao).
    • nums[mid] < nums[end] → end = mid (mid ko include karke left mein jao).
  5. Jab start == end, return nums[start].
// Optimal: Binary Search on Rotated Array - O(log n)
class Solution {
public:
    int findMin(vector<int>& nums) {
        int start = 0;
        int end = nums.size() - 1;

        while (start < end) {           // Note: start < end
            int mid = start + (end - start) / 2;

            // Drop exists between mid and end -> min on right
            if (nums[mid] > nums[end]) {
                start = mid + 1;
            }
            // Right side sorted -> min at mid or left
            else {
                end = mid;              // mid ko discard nahi karte!
            }
        }
        return nums[start];         // start == end == min index
    }
};

Edge Cases & Gotchas
  • Array not rotated: Jaise [11,13,15,17] — nums[mid] < nums[end] hamesha true hoga, end left ki taraf shift hota rahega, finally start=0 pe minimum milega.
  • Single element: [1] — loop nahi chalega, directly nums[0] return hoga.
  • while(start < end) vs while(start <= end): Yaha start < end use karte hain kyunki end = mid karne pe mid ko include karte hain. <= use karne se infinite loop ho sakta hai.
  • end = mid (not mid - 1): Kyunki mid khud minimum ho sakta hai, isliye usse discard nahi karte.

Problem 2 : Find Minimum in Rotated Sorted Array II [LeetCode 154]

Problem Statement Given: Ek array sorted ascending order mein tha, phir rotate kar diya gaya. Is baar array mein duplicate elements ho sakte hain.
Task: Array ka minimum element return karna hai.
Difference from LC 153: Yaha unique elements ki guarantee nahi hai.
Examples
InputOutputExplanation
[1,3,5]1Array rotate nahi hua. First element minimum.
[2,2,2,0,1]0Duplicates ke saath 0 minimum hai.
[3,3,1,3]1Duplicates hone ke baad bhi 1 minimum.
[10,1,10,10,10]1Duplicates search direction ambiguous banate hain.
[2,2,2,2,2]2Sab same hain. Koi bhi element minimum.

Core Intuition

Intuition / Insight LC 153 ka logic yaha bhi kaam karega, lekin ek new problem aa jata hai: duplicates. Jab nums[mid] == nums[end] hota hai, toh hum decide nahi kar sakte ki minimum kis side mein hai.

Example: [3,3,1,3] mein mid=1 (value 3), end=3 (value 3). Kya minimum left mein hai ya right mein? Pata nahi! Dono side same value hai.

Solution: Jab nums[mid] == nums[end], toh ek duplicate safely remove kar do — end--. Minimum element remove nahi hoga kyunki agar nums[end] minimum tha, toh nums[mid] bhi wahi value ka tha (equal hai na!).
Duplicate Ambiguity Example
3 3 1 3
mid = 1 (value 3)  |  end = 3 (value 3)  |  Equal! Direction ambiguous.
↓ Safely shrink from right: end--

Approach 1 : Brute Force (Linear Scan)

Workflow / Algorithm Same as LC 153 — traverse karke minimum nikaal lo. O(n) time, O(1) space. 
// Brute Force: O(n)
class Solution {
public:
    int findMin(vector<int>& nums) {
        int minimum = nums[0];
        for (int num : nums) {
            minimum = min(minimum, num);
        }
        return minimum;
    }
};

Approach 2 : Optimal (Modified Binary Search)

Key Insight Three cases compare nums[mid] with nums[end]:
  • nums[mid] > nums[end] → Minimum right side mein. start = mid + 1.
  • nums[mid] < nums[end] → Minimum mid ya left mein. end = mid.
  • nums[mid] == nums[end] → Ambiguous! Ek duplicate safely hatao. end--.
Algorithm Steps
  1. start = 0, end = n - 1 initialize karo.
  2. while (start < end) loop chalao.
  3. mid = start + (end - start) / 2 calculate karo.
  4. Three-way comparison:
    • nums[mid] > nums[end] → start = mid + 1.
    • nums[mid] < nums[end] → end = mid.
    • nums[mid] == nums[end] → end-- (shrink by 1).
  5. Return nums[start].
// Optimal: Modified Binary Search - O(log n) avg, O(n) worst
class Solution {
public:
    int findMin(vector<int>& nums) {
        int start = 0;
        int end = nums.size() - 1;

        while (start < end) {
            int mid = start + (end - start) / 2;

            // Case 1: Drop on right side
            if (nums[mid] > nums[end]) {
                start = mid + 1;
            }
            // Case 2: Min at mid or left
            else if (nums[mid] < nums[end]) {
                end = mid;
            }
            // Case 3: Duplicate! Shrink safely
            else {
                end--;                  // Remove one duplicate
            }
        }
        return nums[start];
    }
};

Complexity Analysis

  • Why end-- and not start++? Kyunki nums[end] duplicate ho sakta hai, lekin nums[mid] bhi same value ka hai. end-- se minimum safe rehta hai kyunki same value ka ek aur copy bachi hai.
  • Worst case O(n) kyun? Jab sab elements equal hain, toh har iteration mein sirf 1 element shrink hota hai. Binary Search degrade ho ke linear search ban jata hai.

Problem 3 : Search in Rotated Sorted Array [LeetCode 33]

Problem Statement Given: Ek integer array ascending order mein sorted tha, phir kisi unknown pivot par rotate kar diya gaya. Array mein unique elements hain.
Task: Given target value ka index return karna hai. Agar target exist nahi karta, toh -1 return karna hai.
Constraint: Solution O(log n) time mein chalna chahiye.
Examples
InputTargetOutputExplanation
[4,5,6,7,0,1,2]04Target 0 index 4 par hai.
[4,5,6,7,0,1,2]3-1Target exist nahi karta.
[1]0-1Single element, target nahi hai.
[6,7,8,1,2,3,4,5]35Target right sorted half mein hai.

Core Intuition

Intuition / Insight Rotated sorted array mein har iteration pe ek half hamesha sorted rehta hai. Yeh property sabse important hai!

Jaise [4,5,6,7,0,1,2] mein agar mid = 3 (value 7) hai, toh left half [4,5,6,7] sorted hai. Agar mid = 5 (value 1) hai, toh right half [0,1,2] sorted hai.

Strategy: Pehle sorted half identify karo. Phir check karo ki target us sorted half mein lie karta hai ya nahi. Agar haan, toh usi half mein search karo. Nahi toh dusre half mein jao.
Identifying Sorted Half
4 5 6 7 0 1 2
LEFT HALF SORTED (nums[start] <= nums[mid])
6 7 8 1 2 3 4 5
RIGHT HALF SORTED (nums[mid] < nums[start])

Approach 1 : Brute Force (Linear Search)

Workflow / Algorithm
  1. Array ko traverse karo.
  2. Har element ko target se compare karo.
  3. Match milne pe index return karo.
  4. End tak nahi mila toh -1 return karo.
// Brute Force: O(n)
class Solution {
public:
    int search(vector<int>& nums, int target) {
        for (int i = 0; i < nums.size(); i++) {
            if (nums[i] == target) {
                return i;
            }
        }
        return -1;
    }
};

Approach 2 : Optimal (Modified Binary Search)

Key Insight Har iteration pe do cases:
  • Left half sorted (nums[start] <= nums[mid]):
    • Target [nums[start], nums[mid]) range mein hai? → end = mid - 1.
    • Nahi? → start = mid + 1.
  • Right half sorted (else case):
    • Target (nums[mid], nums[end]] range mein hai? → start = mid + 1.
    • Nahi? → end = mid - 1.
Note: Pehle nums[mid] == target check karte hain — agar match ho gaya toh directly return.
Algorithm Steps
  1. start = 0, end = n - 1 initialize karo.
  2. while (start <= end) loop chalao (classic BS).
  3. mid = start + (end - start) / 2 calculate karo.
  4. Base case: nums[mid] == target → return mid.
  5. Left half sorted? nums[start] <= nums[mid]:
    • target >= nums[start] && target < nums[mid] → end = mid - 1 (left mein search).
    • Else → start = mid + 1 (right mein search).
  6. Right half sorted (else):
    • target > nums[mid] && target <= nums[end] → start = mid + 1 (right mein search).
    • Else → end = mid - 1 (left mein search).
  7. Loop end pe -1 return karo (target nahi mila).
// Optimal: Modified Binary Search - O(log n)
class Solution {
public:
    int search(vector<int>& nums, int target) {
        int start = 0;
        int end = nums.size() - 1;

        while (start <= end) {          // Classic BS: start <= end
            int mid = start + (end - start) / 2;

            // Target found!
            if (nums[mid] == target) {
                return mid;
            }

            // Left half is sorted
            if (nums[start] <= nums[mid]) {

                // Target lies in left sorted half
                if (nums[start] <= target && target < nums[mid]) {
                    end = mid - 1;
                }
                else {
                    start = mid + 1;
                }
            }
            // Right half is sorted
            else {

                // Target lies in right sorted half
                if (nums[mid] < target && target <= nums[end]) {
                    start = mid + 1;
                }
                else {
                    end = mid - 1;
                }
            }
        }
        return -1;                   // Target not found
    }
};

Edge Cases & Gotchas
  • Single element: [1] with target 1 → mid=0, match, return 0. Target 0 → loop ends, return -1.
  • Target at pivot: [4,5,6,7,0,1,2] target=7 → Iteration 1 mein hi mid=3 pe match.
  • Array not rotated: [1,2,3,4,5] → Left half hamesha sorted, normal binary search behave karega.
  • Boundary conditions: target == nums[start] ya target == nums[end] ka case properly handle karo. Conditions mein <= aur < carefully use karo.
  • Why start <= end? Is problem mein target exact index pe milna chahiye. start < end use karne se last element miss ho sakta hai.
Interview Tips / Follow-up Questions
  • Q: Is problem ko recursive binary search se kaise solve karoge?
    A: Base case: start > end → return -1. Mid pe target check, phir sorted half identify karke recursive call.
  • Q: Agar duplicates allowed ho toh?
    A: LeetCode 81 — jab nums[start] == nums[mid] == nums[end], toh start++ aur end-- karo. Worst case O(n) ho jayega.
  • Q: Pivot index kaise nikaaloge?
    A: Minimum element ka index hi pivot hai. LC 153 ka logic use karo.
  • Q: Agar multiple rotations ho?
    A: Array ko kitni baar bhi rotate karo, structure same rehta hai — ek sorted half hamesha milta hai.

Common Mistakes (Red Flags)
• Min problems mein <= mat use karo
• Search problem mein < mat use karo
• Min mein end = mid (mid discard nahi)
• Search mein end = mid - 1 (normal BS)
• mid = start + (end-start)/2 (overflow safe)
